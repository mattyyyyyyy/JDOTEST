## TL;DR

> **车机电商的一级分类应该按"用车场景动作"组织，而不是按"商品品类"组织。**
> 司机踩刹车进服务区时，心智里冒出来的是"加油 / 吃饭 / 买咖啡"，不是"我要逛数码区"。**把场景顶上去，品类才用得上。**

---

## 1. 两种 IA 拆法的对比

### 1.1 品类型 IA（传统电商照搬）

**代表样本**：JDO 当前 v0.4 / 大多数车厂自营商城（蔚来 / 小鹏 / 理想）

```
一级 = 商品归属
  ├─ 推荐 / 秒杀（运营位）
  ├─ 车品 / 数码 / 食品
  ├─ 服饰 / 运动 / 出行
  └─ 图书 / 母婴 / 服务
```

**心智模型**：用户站在虚拟货架前 → 选品类 → 进商品 → 决策

### 1.2 场景型 IA（车机原生）

**代表样本**：车机商店 v0.2.0-foundation（Appetize 实机观察）

```
一级 = 用车场景动作
  ├─ 加油充电（在路上 / 服务区）
  ├─ 洗车保养（停车场 / 4S）
  ├─ 周边餐饮（导航顺手）
  ├─ 旅行服务（长途 / 跨城）
  └─ 汽车用品（兜底品类入口）
```

**心智模型**：司机站在仪表盘前 → 我现在在做什么 → 一个动作完成

## 2. 为什么车机要换成场景型

### 2.1 车机不是手机的"横屏放大版"

[竞品报告](./competitor-analysis.md) §我们应该规避的 3 件事 第 1 条已经写过：

> 不要做「手机商城的横屏放大版」：淘宝/京东那种密集瀑布 + 无限下拉，对车机来说是反模式（视觉扫描成本高、单次操作时间长）。**信息密度反而要降低。**

把这条结论再推一步：**降低信息密度的本质就是降低分类树的复杂度**。12 类 → 5 类不是简单"砍 7 类"，是切换分类轴。

### 2.2 决策入口不一样

| 媒介 | 决策入口 | 一次会话目标 |
|---|---|---|
| **手机电商** | "我要买 X" → 主动检索 | 比价、凑单、慢决策 |
| **车机电商** | "我现在在 Y 场景" → 被动触发 | 单一动作、快决策 |

手机用户主动开 App、有时间逛、品类树是导航工具。车机用户被动需求（饿了、油表见底、副驾说"买点零食"），有 NHTSA 12s 累计交互上限，**品类树是阻碍**。

### 2.3 行车态下的强降级要求

[设计系统 §行车态降级](../design/design-system.md) 与 [ADR-0004 行车态车速数据源协议](../decisions/ADR-0004-driving-state-source.md) 都要求：单步 ≤ 2s、全流程 ≤ 12s。

12 类品类树需要"扫一眼 → 决定我属于哪一类 → 点 → 进二级 → 浏览"——光"扫一眼分类"就接近 2s 预算。**5 类场景型分类**因为对应的是司机已经有的动作意图，几乎不需要"分类决策"步骤。

### 2.4 时空数据是车机独有的杠杆

车机有手机没有的两个信号：

- **位置**（GPS + 高德 / 百度 POI · 实时）
- **车况**（剩余里程 + 油量 + 充电状态 · 通过 JS Bridge）

这两个信号天然适合驱动场景型推荐（"前方 3 km 服务区 · 8 家可自提"），但完全不适合驱动品类型推荐（"前方 3 km · 数码品类有 200 件商品"是个荒谬的句子）。**IA 不切到场景，这些数据就用不上。**

## 3. 反方观点 + 反驳

### 3.1 "品类树搜索效率更高"

**反方**：消费者已经习惯了京东淘宝的 12+ 类目，重学成本高。

**反驳**：（a）车机不是消费者主动选择的工具，是车厂塞进来的，没有"用户习惯"包袱；（b）场景型 IA **不排斥品类**，只是降级到二级 —— "汽车用品"本身就是兜底品类入口，"全部商品"也可以保留。

### 3.2 "场景太少不够覆盖长尾"

**反方**：5 类场景能 cover 全部用车需求吗？长尾商品（如车主想买本书）放哪？

**反驳**：（a）长尾本来就不是车机电商的核心场景（车主真想买书会用手机）；（b）"汽车用品"作为兜底类已经接住 80% 非场景类需求；（c）需要时可加 1–2 个场景类（"应急救援"、"长途旅行"）补强。

### 3.3 "我们已经做了 12 类 mockup，改回来要重做"

**反方**：mockups 已经画好，design tokens 已经定，IA 重切代价大。

**反驳**：（a）现在仍是 Demo 阶段，mockups 也是 Demo，**没有真实用户、没有真实交易、没有车厂集成依赖**，是 IA 重切窗口期最大的时候；（b）design tokens 不受影响（颜色 / 字体 / 间距 / 圆角都不变），受影响的只是 `mockups/category.html` 和 `mockups/jdo-pencil/screens/mall-home.jsx` 的 rail 内容 + 后端 `categories` 表的种子数据。

## 4. 提议的新 IA

### 4.1 一级分类（场景型 · 6 个）

| ID | 名称 | 主要场景触发 | 典型商品 |
|---|---|---|---|
| `refuel` | 加油充电 | 油表见底 / 进服务区 / 充电桩附近 | 加油卡 / 充电预约 / 玻璃水 |
| `wash` | 洗车保养 | 停车后 / 进 4S 附近 | 洗车券 / 美容套餐 / 玻璃水 / 内饰清洁 |
| `dining` | 周边餐饮 | 导航中 / 服务区 / 用餐时段 | 服务区餐饮 / 自提咖啡 / 零食 |
| `travel` | 旅行出行 | 长途 / 跨城 / 工作日通勤外 | 高速通行卡 / 酒店 / 景区门票 |
| `parts` | 车品配件 | 兜底品类（手机电商习惯） | 香薰 / 行车记录仪 / 充电宝 / 脚垫 |
| `sos` | 应急救援 | 故障 / 事故 / 故障码报警 | 道路救援 / 拖车 / 维修预约 |

### 4.2 二级分类

二级保留原 12 类的细粒度（数码 / 食品 / 服饰 / …），但**只作为"车品配件 → 全部商品"下的浏览入口**，不出现在一级 rail。

### 4.3 首页 hero

- 不再是"秒杀倒计时"，而是 **"基于位置 + 时段 + 车况" 的场景化推荐 4 张卡**
- 例：
  - 「前方 3 km · 服务区 · 8 家自提点」
  - 「续航剩 78 km · 附近充电桩 3 处」
  - 「中午 12:14 · 附近车主常订午餐」
  - 「上次保养已 5800 km · 推荐洗车套餐」

## 6. 后续可调研

- **场景与商品的多对多映射**：一瓶玻璃水可以挂在"加油充电""洗车保养""车品配件"3 个场景下，需要打 tag 而不是单一归属。设计 schema 时需要注意。
- **场景的位置数据来源**：高德 POI API vs 百度地图 API vs 自建 —— 影响 [ADR-0005 部署方案](../decisions/ADR-0005-deployment-strategy.md) 的供应商成本
- **应急救援场景的合规边界**：是导购 + 跳第三方？还是接入 122 / 平安道路救援？影响法律责任划分
- **第二轮竞品**：抓更多真机数据（车机商店 v0.2.0 只看到一级），尤其是蔚来 / 小鹏 / 比亚迪的车机商城横屏截图
